# GOAL · Day 01 AI 视频运营流水线

## 北极星

让高频视频运营团队不再在 AI 对话、飞书、文件系统、手机和平台后台之间搬运同一条视频：任何一条视频都能从 Context 走到脚本分镜、成片、多账号发布、数据回流和下一轮裂变，而且身份、版本和血缘始终不断。

## 用户承诺

用户进入产品后，不需要先理解工作流引擎、平台 API 或数据模型。他只需要看到：

- 现在有哪些视频；
- 每条视频做到哪一步；
- 下一步可以点什么；
- 发到了哪些账号；
- 哪些流量高、哪些成交高；
- 点击哪条视频可以继续裂变。

产品不是“AI 分析报表”，而是每天工作的主操作台。

## 不可变产品决定

1. `Video` 是最小功能单位，从创建起贯穿全部阶段；MP4 只是它的一个资产。
2. 账号是管理、Context 和发布维度；平台只是账号属性。
3. 一条视频可投给多个账号，每个账号产生独立 `Publication` 和独立数据。
4. 同一成片跨平台仍是一条视频；实际内容改变才创建派生视频。
5. Context 是任意来源组成的快照，不是要求用户填写的巨大字段表。
6. 商品是可选 Context；账号既能发带货视频，也能发非带货视频。
7. 脚本和分镜必须有内置生产能力，也必须允许导入现成结果和替换外部模块。
8. 数据首先服务视频排序、标识和裂变，不建设脱离生产流程的报表中心。
9. 飞书是导入、导出和同步目标，不是产品唯一事实源。
10. 外部脚本、剪辑、发布和数据工具都可以替换，视频记录和血缘不能丢。
11. 一批脚本先产生可比较的候选；只有用户选中的候选才成为正式 `Video`，未选候选只在所属批次内以待选稿呈现。
12. 发布前只能判断脚本是否合规且值得测试；“高转化”必须由真实发布、点击和成交数据证明。
13. 用户可以显式推进“需修改”稿，但选择不会清除风险；结构状态、声明与待补证据必须随正式脚本保存并继续可见。

## 核心对象

```text
Workspace
├─ AccountGroup
├─ Account(platform, connection, account context)
├─ ProductSource(optional reusable context)
├─ Batch(primary browsing and operation collection)
└─ Video
   ├─ ContextSnapshot[]
   ├─ ScriptArtifact[]
   ├─ StoryboardArtifact[]
   ├─ MediaArtifact[]
   ├─ Publication[]
   │  ├─ MetricSnapshot[]
   │  └─ CommentSnapshot[]
   └─ Lineage(parent, children, variation note)
```

批次是工作台一级浏览与批量操作单位，但不会吞并视频身份；`Video` 仍独立贯穿产物、发布、数据与血缘。账号与视频是多对多关系。

## 主流程

### A. 从零创建

1. 用户点击“新建视频”或“批量生成”。
2. 可选择商品，也可不选；添加一句主 Context、参考链接或已有素材，并决定脚本数量。
3. 系统先把商品事实与 Context 编译成创作简报，一次生成多个完整叙事角度不同的脚本候选。
4. 用户比较、展开和修改候选，只做一次多选；结构检查不通过的候选不能伪装成高质量结果。
5. 选中的候选创建正式视频与脚本/分镜 v1，进入待成片；演员、声音、画幅和制作方式后置。
6. 目标账号到发布阶段再选择，一条视频可建立多个独立发布任务。

### B. 接入既有流程

1. 导入飞书、文档或外部 AI 已经完成的脚本和分镜。
2. 系统创建视频、保存原始来源并形成正式版本。
3. 后续成片、发布、数据和裂变与内置生成完全一致。

### C. 成片和多账号发布

1. 上传成片或由外部制作模块回传成片。
2. 选择一个或多个目标账号及排期。
3. 每个账号生成独立发布任务；平台适配器推进状态并返回平台视频编号和链接。
4. 失败只影响对应发布记录，不回滚视频和其他账号的成功记录。

### D. 数据回流与裂变

1. 数据按发布记录定时形成快照，不覆盖历史观察值。
2. 评论按发布记录回流，保留平台评论编号、正文、互动数和获取时间；可以作为下一轮 Context。
3. 视频列表支持按播放、成交、发布时间和状态排序，并显示流量高、成交高、双高或待观察。
4. 用户点击“裂变”创建子视频；系统继承父视频 Context、脚本、分镜、素材引用和数据摘要。
5. 用户只需说明本轮变化，或选择换钩子、人物、场景、口播、卖点顺序等常用入口。
6. 子视频重新进入脚本/分镜阶段，并在血缘视图中可追溯。

## 通过与不通过

### 通过

只有当用户能在真实工作台完成“创建/导入视频 → 脚本分镜 → 成片 → 多账号发布记录 → 数据 → 裂变”，并同时通过零密钥 Demo 和上方“真实验证”清单全部条目，Day 01 才能进入 `verified`。

### 不通过

出现任一情况都不能宣称完成：

- 只有 UI，没有真实状态和产物；
- 只有脚本生成，没有成片、发布、数据和裂变关系；
- 只有数据看板，不能回到视频、脚本和下一批生产；
- 发布和回流只用 mock，却对外写成真实打通；
- 裂变只是复制文本，没有父子关系和 Context 继承；
- 为了接平台把 token、账号或本机路径写进仓库；
- 依赖私有工具才能跑 Demo。

## 明确不做

- 自研大模型、视频生成模型、剪辑器、转录器或平台发布协议；
- 在 Day 01 内解析图片/视频画面；首版保留可追溯地址，后续交给可替换的提取器或制作模块；
- 批量扫描平台账号全部历史视频；首版支持已知平台编号/链接关联，批量迁移走后续平台导入器；
- 在 Day 01 建插件市场、计费、企业权限、无限层级组织树；
- 预先拆完所有商品、账号和视频字段；
- 自动判断“必爆”或承诺提高 GMV；
- 同时适配所有内容平台；
- 在 Day 01 调用或集成飞星 / TikTok CLI；其文档只用于校准平台适配器边界；
- 用大量人工确认弹窗替代自动推进。

## 成功标准

产品成功不是生成了一段漂亮脚本，而是团队不再需要用飞书手工串联同一条视频，并且任何高表现视频都能在一个动作后进入下一轮生产，同时保留完整来源和结果。
