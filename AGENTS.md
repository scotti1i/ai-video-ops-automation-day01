# AGENTS.md · Day 01 AI 视频运营流水线

动代码前先读本仓库 `app.toml`、`GOAL.md`、`PRODUCT.md`、README、`research/upstreams.md` 和 `design.md`。本仓库可独立运行，不依赖其他目录。

## 业务合同

- 角色：每天生产、交付和发布数十条自制视频的运营团队；组织可能是商家、品牌方或代运营，不在产品里硬拆角色。
- 触发：团队从账号、商品、对标视频或已有脚本启动一条或一批视频，并持续推进到发布、数据回流和裂变。
- 输入：任意 Context（对话、文字、链接、文件、图片、参考视频）、已有脚本分镜、成片、账号与排期。
- 输出：带版本和血缘的视频记录、脚本分镜、成片资产、多账号发布记录、平台数据快照和裂变子视频。
- 人工判断点：只保留真实发布等不可逆外部动作的明确发起；不把常规脚本生成、数据同步和裂变判断甩回用户。
- 明确不做：自研视频模型、自研剪辑器、自研平台协议、巨大商品字段系统、把报表或聊天窗当成产品主体。

以上任一项不清楚，先回需求池补证据，不开始堆功能。

## 命令合同

所有命令以 `app.toml` 为准：`doctor / demo / check / run`。改代码后先跑 demo 和 check，再看真实产物。

## 硬规则

1. 样例数据和真实验证分开；mock 不能写成已落地。
2. 关键数字只从结构化输入计算一次，模型不能编数字。
3. 平台能力拿不到就降级为导入/导出或人工确认，不绕登录和风控。
4. 输出只写 `output/` 或 `.local/`；两者都不进 git。
5. UI 开发前 `design.md` 必须有对标、色板、字体和 3 个禁止项，并已确认。
6. 函数不超过 50 行，文件不超过 800 行，三层嵌套封顶。
7. 发布前用真实脱敏输入跑一遍，并把证据写进 `evidence/`。
8. 先比较本地资产、GitHub 轮子和最简脚本；只引入有 License 且能锁版的依赖。
9. UI 默认浅色白底、海军蓝、苹果系无衬线；禁紫色渐变、persona 卡墙和手机溢出。
10. `Video` 是最小功能单位；账号是管理与发布维度，平台只是账号属性。一条视频可有多条独立发布记录。
11. 内置脚本/分镜生产、导入既有成果和外部模块接入必须同时存在，任何一种方式都要落成同一份版本化产物。
12. Day 01 只实现真实 YouTube 连接器；飞星 / TikTok CLI 不得调用，其他平台只保留合同、能力声明和模拟实现。
